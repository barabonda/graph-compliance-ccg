# 평가 기준 대응 정리

이 문서는 JB금융그룹 AI Agent 서비스 평가 기준에 맞춰 발표 자료에서 반드시 보여줘야 할 논리와 증거를 정리한다.

## 1. 주제 적합성 및 문제 정의

심사위원 질문:

> 왜 금융광고 준법심사에 AI Agent가 필요한가?

답변 방향:

- 금융광고 심사는 단순 문구 검사가 아니라 상품군, 채널, 금소법/시행령/감독규정, 협회 심의기준, 상품설명서 사실, 필수 고지를 함께 봐야 하는 구조적 판단이다.
- 마케팅 초안은 빠르게 늘어나지만 준법 검토는 사람이 법령, 상품문서, 고지 체크리스트를 오가며 확인해야 해 병목이 생긴다.
- 위반 문구를 놓치면 제재·민원·브랜드 리스크가 크고, 반대로 과도하게 막으면 출시 속도와 마케팅 효율이 떨어진다.

장표에 넣을 증거:

- 광고 문구 예시: `누구나 조건 없이 최고 연 5.0% 확정`
- 필요한 확인 항목: 우대조건, 가입대상, 금리 기준, 예금자보호 한도, 심의필, 금소법 광고규제
- 현재 콘솔 캡처: 문구별 리스크와 수정안

## 2. 금융 업무·사업 연계성 및 고객 가치

심사위원 질문:

> JB금융 업무 안에서 실제로 어디에 쓰이는가?

답변 방향:

- 사용자는 마케팅 담당자와 준법 담당자다.
- 마케팅 담당자는 초안을 올리고, 준법 담당자는 위험 문구·필수 고지·수정안을 확인한다.
- 은행 예금/적금 광고에서 시작해 대출, 투자, 보험, 카드, 이벤트 광고로 확장할 수 있다.
- 고객 관점에서는 오인 가능성이 낮은 광고를 보게 되고, 내부 관점에서는 검토 시간과 누락 리스크를 줄인다.

기대 가치:

- 검토 초안의 1차 분류 자동화
- 위험 문구와 누락 고지 빠른 발견
- 상품설명서와 광고 문구의 사실 불일치 탐지
- 근거 조항과 수정 이력 보존

## 3. AI Agent 설계 및 기술적 구현 가능성

심사위원 질문:

> 그냥 RAG 챗봇과 무엇이 다른가?

답변 방향:

- RAG는 관련 문서를 찾는 데 강하지만, 어떤 조항을 지금 광고에 적용해야 하는지, 어떤 고지는 이번 상품군에 필요한지, 어떤 기준은 제외해야 하는지까지 안정적으로 판단하기 어렵다.
- 이 서비스는 그래프와 게이트가 먼저 판단 범위를 좁힌다.
- LLM은 모든 법령을 자유롭게 해석하는 것이 아니라, 정리된 Evidence Window 안에서 CU별 판단과 수정안을 생성한다.

기술 구조:

```text
광고 초안
→ Hierarchical Context Graph
→ PolicyHypernym / CU 후보 검색
→ Product Fact Graph 대조
→ Disclosure Applicability Gate
→ Cross-encoder / LLM rerank
→ LLM Judge
→ Exception / Mitigation Review
→ Routing + Revision + Audit
```

구현 가능성을 보여줄 증거:

- FastAPI backend
- Next.js/React console
- Neo4j Policy Graph
- ProductDocument/ProductFact extraction
- streaming review pipeline
- audit trace and run history

## 4. MVP 완성도 및 검증 가능성

심사위원 질문:

> 실제로 동작하고 검증 가능한가?

답변 방향:

- MVP는 광고 입력, 상품 선택, 실시간 분석, 문구별 리스크, 상품 사실 대조, 법적 근거, 필수 고지, 수정안, 실행 기록까지 연결되어 있다.
- 평가 지표는 단순 정확도보다 금융 준법 업무에 맞춘 recall과 evidence grounding을 본다.
- 정상 광고가 과도하게 reject되는 overblocking도 별도 관리한다.

핵심 지표:

- `ProductFact extraction rate`: 상품문서에서 사실을 추출한 비율
- `ClaimFact comparison accuracy`: 광고 주장과 상품 사실 비교 정확도
- `Evidence grounding rate`: 판단에 근거 조항/상품문서가 붙은 비율
- `CUPlan recall`: 위험 문구에 관련 ComplianceUnit이 최소 1개 연결되는 비율
- `micro-F2`: 위반 누락을 줄이기 위한 recall 중심 지표
- `overblocking_rate`: 정상 문안이 reject로 가는 비율
- `cu0_rate`: 정책 매칭 실패 비율

장표 표현:

> "정확도 하나가 아니라, 위반 누락·과차단·근거 연결·상품 사실 대조를 함께 측정합니다."

## 5. 혁신성·확장성 및 운영 리스크 관리

심사위원 질문:

> 기존 업무 방식보다 무엇이 새롭고, 운영 리스크는 어떻게 줄이는가?

차별성:

- 법령 검색이 아니라 광고 문구와 상품 사실, 필수 고지, 소비자 인상을 함께 정렬한다.
- `누구나`, `조건 없이`, `확정`, `보장` 같은 표현을 독립 키워드가 아니라 parent claim 안의 qualifier로 해석한다.
- 고지가 존재해도 혜택 문구보다 현저성이 낮으면 별도로 진단한다.
- 개별 조항 심사와 전체 인상 심사를 분리해 과소/과대 판단을 줄인다.

리스크 대응:

- 최종 결정은 사람이 한다.
- 모델 판단에는 근거 조항, 상품문서, 고지 상태를 함께 표시한다.
- OpenAI/Claude/로컬 LLM 등 모델 교체 가능 구조를 둔다.
- 민감한 광고 초안은 내부망/로컬 모델 적용 경로로 확장할 수 있다.
- 법령/심의기준 버전과 실행 조건을 audit log에 남긴다.

확장 방향:

- 은행 예금/대출에서 보험, 카드, 투자상품 광고로 확장
- 웹페이지 텍스트에서 SNS, 배너, 유튜브 자막/음성 고지로 확장
- JB금융 계열사별 내부 심의 기준 연결
- 규정 변경 감지와 회귀 테스트 자동화
