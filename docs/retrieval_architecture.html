④ 검색 로직의 3개 함수 (retriever.py)

a) assert_policy_alignment_ready 준비 게이트

그래프가 검색 가능한 상태인지 빠르게 차단. 조건: hypernym ≥ 30, premise ≥ 100, 활성 CU의 hypernym 연결 비율 ≥ 0.8. 하나라도 미달이면 즉시 예외(폴백 없음). retriever.py 72–103

b) policy_context_for_claims 정규화기 입력

정규화 LLM에 넘길 컨텍스트를 모은다. 세 덩어리의 정렬 방식이 서로 다르다:

덩어리정렬한도정규화기가 실제로 쓰는 양
hypernymspriority 정렬 (질의 무관)max(limit,2000)앞 200개만 (normalizer.py 90)
premisesORDER BY 없음 (임의)80앞 80개 (normalizer.py 91)
fragments코사인 유사도 top-4040앞 40개 (normalizer.py 92)

→ 질의 관련도로 랭크되는 건 fragments(40개)뿐. 어휘·전제는 관련도와 무관하게 들어간다. retriever.py 105–145

c) candidates_for_anchor 핵심 후보검색

앵커당 CU 후보를 찾는 본체. 흐름:

  1. 조기 차단: 앵커에 hypernym 제안이 없으면 return [] — 임베딩 안 보고 끝. 157–158
  2. 앵커 텍스트(span+facts+hypernym why) 임베딩 1회. 159–166
  3. Cypher: 임베딩 보유 활성 CU 전부에 대해 vector.similarity.cosine 계산 + hypernym id/name 겹침 계산. 176–195
  4. WHERE 통과 조건: id겹침>0 또는 name겹침>0 또는 (vector≥0.68 그리고 principle≠'제재'). 196–201
  5. 근거 수집: premise(≤6) + premise source(≤4) + 직접근거(≤4). 202–228
  6. Cypher ORDER BY로 거른 뒤 LIMIT 300 → Python에서 점수 재계산·게이트·정렬 후 top 12. 245–275

⑥ 검색 후 게이트/필터 (점수 외 탈락 경로)

점수가 높아도 아래 휴리스틱에 걸리면 조용히 제거된다. 점수 추적에 안 남는다.

scope gate

상품군 토큰 불일치 → suppressed. 예: product_group=deposit인데 CU 텍스트에 '대출/펀드' 토큰이 있고 '예금' 토큰이 없으면 억제. 393–427

operational 필터

'제재·벌점·심의필·협회·준법감시인' 등 토큰 또는 cu_type ∈ {SANCTION, PROCEDURE, ...} → 제거. 430–452

anchor 적격

product_anchor / target_consumer_anchor는 hypernym 겹침이 없으면 전부 탈락(=scope 전용). principle='제재' 탈락. 379–390

LLM rerank

통과한 후보를 cross-encoder식 LLM이 앵커당 ≤5, 총 ≤20개로 재선별. actionable 앵커가 0개로 비면 최고점 1개를 강제 보존. planner.py 100–156

⑦ 병목 · 비효율 진단

심각도 순. 각 항목에 코드 위치와 권장 개선을 표기.

치명 1. 검색 리콜이 단일 LLM 호출에 전부 종속

정규화 LLM이 앵커에 PolicyHypernym을 못 붙이면 candidates_for_anchor가 첫 줄에서 return []. 임베딩이 아무리 가까운 CU가 있어도 검색이 0건이 된다. 벡터 단독 fallback 경로가 없다(쿼리 내부엔 vector 분기가 있으나, 함수가 그 전에 조기 반환).

retriever.py 156–158 · 의존: normalizer.py(LLM) → 실패코드 NO_HYPERNYM_MATCH (router.py 253)

개선: hypernym이 비어도 앵커 임베딩으로 CU 벡터검색을 돌리는 fallback을 추가(이미 있는 vector 분기를 함수 조기반환 위로 끌어올리기). retrieval_probe.py §5가 이 구멍을 실측한다.

치명 2. 근거 창이 얇아 판단이 INSUFFICIENT로 샘 (실측 4/7)

판단 LLM에 들어가는 evidence window는 anchor.facts + CU의 evidence_texts(premise 문장/출처 스니펫, 각 ≤360자, 소수)만 담는다. LegalChunk 원문은 판단 창에 안 들어간다(설계상 provenance). 그래서 판단기가 "필요한 사실/문서가 창에 없다"며 INSUFFICIENT를 반환하는 비율이 높다.

judge.py 56–77 (window) · judge system prompt 99–108 · planner.py 174

개선: CUPlan에 선택된 CU의 legal_evidence_id에 대해 LegalChunk 원문 일부를 on-demand로 끌어와 창에 주입. 또는 premise 스니펫 360자 한도를 상향.

높음 3. 활성 CU 전수 코사인 — 벡터 인덱스 미사용

candidates_for_anchor는 앵커마다 임베딩 보유 활성 CU 전부에 대해 vector.similarity.cosine를 계산한다. Neo4j 벡터 ANN 인덱스(db.index.vector.queryNodes)를 안 쓴다. 비용 O(앵커수 × 활성CU수). 지금 규모(앵커 3)는 무난하지만 CU/광고가 늘면 핫패스가 된다.

retriever.py 176–201

개선: CUEmbeddingProfile.embedding에 벡터 인덱스를 만들고 ANN top-k로 1차 후보를 좁힌 뒤 hypernym 겹침을 재랭킹. probe가 활성 CU 수와 단계 소요시간을 출력한다.

높음 4. 두 개의 서로 다른 점수식 + 죽은 항

Cypher ORDER BY는 0.55·vector + 0.35·overlap + 0.10로 1차 정렬 후 LIMIT 300, Python은 0.55·vector + 0.35·overlap + 0.10·active로 재정렬한다. Cypher의 +0.10은 모든 행에 같은 상수 → 정렬에 무의미한 죽은 항이고, active 가중이 Cypher 1차컷엔 반영되지 않는다. 1차 LIMIT이 최종 식 기준 상위 후보를 미리 잘라버릴 수 있다(현재 규모에선 300이 넉넉해 영향 적음).

retriever.py 245–251 vs 342

개선: 두 곳의 식을 일치시키거나, 1차 컷은 vector만으로 단순화하고 결합점수는 Python 한 곳에서만 계산.

높음 5. 게이트 휴리스틱이 한국어 토큰 부분일치 — 조용한 오탈락

상품군/운영 필터가 연결된 CU 텍스트에 대한 단순 substring 매칭이다. 예금 CU가 본문에 '대출'을 한 번 언급하면 suppressed로 떨어질 수 있고(거짓양성), 반대로 표현이 다르면 운영성 CU를 못 거른다(거짓음성). 탈락이 점수 trace에 남지 않아 디버깅이 어렵다.

retriever.py 409–452

개선: 게이트 결과를 후보에 사유 코드로 기록(억제/탈락 이유)하고, 토큰셋을 그래프 라벨/속성(상품군 태그)로 대체.

중간 6. 어휘 과다 적재 + 하류 절단으로 낭비

policy_context_for_claims는 매 실행 hypernym을 최대 2000개까지 가져오는데(LIMIT max(limit,2000)), 정규화기는 앞 200개만 쓴다(normalizer.py 90). 200개를 넘는 어휘는 DB·전송 비용만 쓰고 버려지며, 동시에 유효 어휘가 priority 정렬상 뒤로 밀려 잘릴 위험이 있다(질의 관련도 정렬이 아님).

retriever.py 119–123 · normalizer.py 90

개선: 어휘를 질의 임베딩 유사도로 1차 랭크해 상위 N개만 적재. 가져오는 한도와 사용 한도를 일치.

중간 7. premise 컨텍스트가 무정렬 LIMIT 80

정규화기로 가는 premise는 ORDER BY 없이 앞 80개. 질의와 무관한 임의의 80개가 들어가 정규화 품질을 흔든다. (유사도 랭크된 fragments 40개와 중복/경쟁)

retriever.py 125–143

개선: premise도 질의 임베딩 유사도로 정렬하거나, fragments로 일원화.

중간 8. 중복 계산 · 임베딩 왕복

Cypher가 이미 profile_vector_score를 계산해 주는데, Python candidate_from_row가 profile.embedding을 통째로 받아 cosine을 한 번 더 계산할 fallback을 둔다(라인 335). 임베딩 벡터를 매 후보마다 와이어로 끌어오는 비용이 있다. 앵커 임베딩도 한 번에 한 앵커씩(embed_many 미사용).

retriever.py 243·332–335 · ccg_embeddings.py 28

개선: Cypher 점수를 신뢰하고 profile.embedding 반환 제거. 앵커 임베딩은 배치(embed_many)로.

중간 9. 그래프 검색 트랙의 낮은 기여도

이번 실행에서 그래프 검색→판단 트랙은 NON_COMPLIANT 0건. 최종 revise는 전적으로 별도 Track B(전체 인상 LLM)가 끌어올린 결과다(router.py 34). 즉 비싼 검색 파이프라인이 이 광고에선 위반 신호를 1건도 못 만들었다 — 리콜(항목 1,2)과 직결된 품질 지표.

router.py 30–43

개선: 항목 1·2를 먼저 고쳐 검색 트랙의 recall을 올린 뒤, Track B는 보조 안전망으로 유지.

⑧ 직접 검색만 격리해서 돌려보기

같이 드린 retrieval_probe.py는 LLM 판단을 빼고 검색 단계만 실제 DB에 돌려, 위 분석의 수치를 직접 확인하게 해줍니다(임베딩 호출만 사용 → 비용 최소).

cd <repo-root>
export PYTHONPATH=examples/graph-compliance-ccg

# 기본: 질의로 자동 hypernym 선택 후 후보검색 점수 분해 출력
python3 examples/graph-compliance-ccg/retrieval_probe.py \
  --workspace-id graphcompliance_mvp_jb_20260530 \
  --text "지난 3년간 조기상환 성공률이 높았던 ELS, 중위험 투자자에게 좋은 선택입니다." \
  --product-group investment \
  --json examples/graph-compliance-ccg/docs/probe_result.json

# 특정 hypernym id를 직접 seed로 주고 싶을 때
python3 ... retrieval_probe.py --hypernym-ids "<id1>,<id2>"

출력: 준비 게이트 결과 / 활성 CU 수 / 정규화 컨텍스트의 세 정렬 / 후보별 (vector·overlap·active·combined·gate) 분해 표 / §5 — hypernym 0개 앵커가 후보 0건이 되는 리콜 구멍 실증. --json으로 저장하면 이 문서 옆에 결과가 남습니다.
